# CLAUDE.md — Life OS

## ⚠️ DATA SAFETY — READ FIRST
- **`vault/` is Sam's REAL notes + journal.** Never bulk-edit, regenerate, mass-rename, or delete vault files. Touch a single note/day only when the task explicitly says so.
- **`data/app.db` is production data** (the NAS copy is the live one). Never wipe or migrate it destructively; schema changes go through `db_init.py` `migrate()`.
- **`vault/` and `data/` are gitignored on purpose** — they are user data synced by Synology Drive, not code. Don't commit them, don't "fix" the ignore.

## Layer 1 — How to work here (behavioral rules)
- **Think before coding**: state your assumptions; if the request is ambiguous, surface the interpretations and pick with reason — don't silently guess.
- **Simplicity first**: the minimum code that solves it; no speculative features, no premature abstraction. If 200 lines could be 50, rewrite it.
- **Surgical changes**: touch only what the task needs; match existing style; don't refactor or "fix" adjacent code unasked.
- **Goal-driven / verify**: every change ends with proof — run `pytest` AND exercise the changed flow on the running server before declaring done.
- **Respect scope**: the cut list below is deliberate; don't reintroduce it without Sam asking.
- **No new deps lightly**: runtime is Flask-only by design; justify anything more.

### In-place updates — no full-page reloads (swept 2026-07-15)
- **Every add/delete/modify patches the ONE node it changed; `reloadSoon()` is a FALLBACK, not the default.** A mutation route returns freshly-rendered markup and the JS swaps that node. Sam asked for this explicitly ("adding and deleting and modifying refreshing the whole page? i dont want that").
- **The server owns all card markup — never hand-build a card in JS**, or a spliced card drifts from a page-load one. `core/web_core.task_card_html(conn, id, surface)` renders one task via the real `_macros.html` macro (`surface`: `week` | `today` | `kcard`); `""` means the task is gone → remove the node. Routes returning `card_html`: `/tasks/new`, `/tasks/<id>/edit|complete|plan|restore` (pass a `surface` form field; `complete` also returns `respawn_html` for a recurrence respawn). `routes/goals.goal_card_html` and `routes/journal._last_entry_html` are the goal/journal twins. Client side: `swapCard(id, html)` / `htmlToNode` / `removeNode`, and `removeWithUndo(el, {onRestore})` — return truthy from `onRestore` to re-insert in place so an Undo costs no reload.
- **`tasks_core.is_pinned(t, today)` is the ONE on-today/pinned predicate** — `tasks_page` buckets the board with it AND `task_card_html` re-derives it per card. If they disagree a swapped card drifts (the `kcard` macro gates its `stale` line on `not t.pinned`). `tests/test_task_lifecycle.py::test_swapped_card_matches_the_page_render` is the tripwire.
- Reloads that REMAIN are deliberate: no node on the page for that id, a Today membership move (hero ↔ week pool — that layout is the page's), a `by_date` goal (its grid sorts by target date, so a new card can't just be appended), or a goals-rail change from the capture bar.

### AI latency — the numbers (measured 2026-07-15, don't re-litigate)
- **`claude -p` has a ~2.9s process floor and 2.5–10s of jitter, even for a 22-char prompt.** A full router call measures **3.4–10.2s (median 5.4s)**. **No AI path can ever feel instant.**
- **Prompt size is NOT the cost** — a 22-char prompt (2.86s median) vs the full 22.2k-char router prompt (3.21s) is inside the noise. **Trimming context buys ~0.3s and costs real capability — don't.** (This was measured after a wrong guess that the context was the problem.)
- **The only lever is calling claude LESS**, which is what `route_deterministic` is for. Deterministic tiers answer in ~1-5ms — 1000×+ faster, e.g. "how many tasks today?" 10.2s → 0.002s, "add reminder 1 min test" 8.1s → 0.001s.
- `router._upcoming_events` is a LIVE Google Calendar call in `build_context` on every capture — cached 60s (`_EVENTS_TTL`); a failure is never cached.
- **Gmail: BATCH, never loop.** 20 `messages.get` calls measure **8.78s issued serially vs 0.46s batched** (`google_client._batch_get`, one `new_batch_http_request`) — 19×. `gmail_search` was the serial shape (1 list + n gets); it's now 1 list + 1 batch (metadata) + 1 batch (bodies) ≈ 3 round trips whatever `n` is. Never add a `for … .execute()` loop over Gmail; `_batch_get` falls back to serial only for a stub/failure.
- **Never walk a widening ladder one round trip at a time.** Every "search, and widen if it misses" ladder had this shape and it was ~2/3 of the pre-seed: Gmail spent **3.8s over 5 sequential searches** (4 always-empty) and Dropbox **5.6s over 8**. Every rung is derivable up front, so probe/run them CONCURRENTLY and take the most specific non-empty (`core.text.first_non_empty`; Gmail uses `gmail_probe` — ~5 quota units vs a full search's ~130 against a 250/s ceiling — then fetches only the winner). **`gather()` measured 9.3s → 1.9s**; a clean question end-to-end **15.3s → ~9s**, now dominated by the irreducible `claude -p` call.
- **`search_documents` is NETWORK-bound, not local** — it unions Dropbox's API. It sat on the main thread ("it needs the conn") and cost **6.7s of a 9.3s gather**, overlapping nothing. It now runs in the pool with its OWN connection (`db.db_path_of` + `connect`; WAL allows concurrent readers). Only genuinely cheap SQLite reads (`_CONN_SOURCES` = facts/tasks/goals) stay on the calling thread.
- **The floor is ONE `claude -p` call (~5-6s).** After the above, that IS the latency. The only way under it is answering deterministically (`answer_from_facts`) — but see the facts-cache caveat below before reaching for that.

### Retrieval — why "I couldn't find it" happened (measured 2026-07-15 on the real mailbox, don't re-litigate)
Sam asked for his August flight booking ref and his cruise date. Both were in Gmail; both answered "not found". **None of the plausible culprits were the cause** — the retrieval worked, the model was fine, and each wrong guess cost real time:
- **NOT the model, NOT the prompt.** Handed five emails about bedding and asked for a cruise, it correctly said it saw no cruise. It was answering the evidence it got.
- **NOT filename/content indexing.** `document_roots` is unset and Dropbox holds none of this — "Searching your docs" only ever searched the vault. An FTS content index over local documents was built here and **reverted**: it indexed the wrong corpus. Don't rebuild it without first checking `document_roots` is actually populated.
- **NOT vocabulary** (teaching it "cruise line X = a cruise"). Gmail's own content search finds the cruise line's name → the sailing-confirmation email instantly, with the date in the subject. Never add a brand/category vocabulary list to a prompt — it's whack-a-mole and it wasn't the bug.
- **THE CAUSE: rank truncation.** Gmail ranks **messages, not conversations**. Six replies under one long admin thread (spread over FOUR threadIds, so per-thread dedup does NOT save you — and each subject carried a sender-appended suffix, so subject-grouping doesn't either) pushed the answer to **rank 7, past `n=5`**. It was never retrieved.
- **The fix is visibility, not a bigger `n`.** `gmail_search` returns EVERY candidate (up to `n * _CANDIDATE_FANOUT`); `n` rations only the costly `body` fetch, dealt round-robin across conversations. A booking's date/ref is usually in the SUBJECT, so headlines alone often answer it. Hits carry `total` (Gmail's `resultSizeEstimate`) and `_evidence_text` prints "showing X of ~Y" — **silent truncation is what turns "not shown" into "doesn't exist", and it sounds exactly like a true negative.** Any new evidence source must say when it truncated.
- **Tuning a constant to one measured example is the trap** (an `n=8` that "fixed" the cruise was rejected for this): the next question has nine crowding threads. Fix the structure so the constant stops mattering.
- **Widening is LOSSY and can only ever know emptiness, never relevance — so it must ANNOUNCE itself.** Same failure one layer down: asking for the "booking number for my fight in August" (Sam's typo) matched nothing, so the ladder dropped **`august`** — the one word that mattered — kept the typo, widened to bare `"booking"`, and got 201 unrelated emails. The model answered from rank 1 and named a **September** trip as the August one: fast, sourced, and wrong. Nothing in the evidence said the query had been reduced. `_gmail_hits` now tags hits with `asked`/`query` and `_evidence_text` prints a ⚠ when they differ, which is what makes the model verify or re-search instead of trusting rank 1. **Any lossy step — truncation, widening, a failed connector — must say so; silent degradation is indistinguishable from a real answer.**
- **Gmail's `resultSizeEstimate` is a capped estimate (201 for everything) — use it as a boolean, never a ranking.** `gmail_probe` returns 0 for "Gmail says none" and **None for "the probe failed"**; callers must not collapse them (None ⇒ fall back to walking the ladder for real).
